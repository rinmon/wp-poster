# WordPress 記事予約投稿ワークフロー

## 概要

ユーザーから記事テキストを受け取り、AI画像を生成してWordPressに予約投稿する。

## 画像方針（重要）

**アイキャッチ（1枚目）**：可能な限り Official・一次情報（公式SNS、プレスリリース等）を検索して活用する。**本文2枚目以降**：**直前の見出し・段落の主題と意味的に一致**するビジュアルに限定する。企業の**汎用OGP・アイコン・プロフィールバナー**を、枚数や差し替え目的だけで並べない。

**そのセクションに即した実写が無い・取得できない・著作権上つかえない場合**は、無関係な写真で埋めず、**Nano Banana**／**Nano Banana Pro**／Cursor の **`GenerateImage`** 等で**その節の内容に沿ったイメージ**を生成し、`drafts/` に保存して IMAGE_BLOCK に入れる（※AI明記・`取得区分: 生成AIイメージ`）。**AI生成画像のみに頼り切る**のではなく、**関連性のない実写より関連性のあるAI**を選ぶ。

- アイキャッチ画像（img_001）：公式サイトの画像、または記事テーマを表すイメージ／生成
- 本文画像（img_002以降）：**各セクションの論点に対応**したビジュアル。**本文は最低3枚分配置**
- 画像数の目安：**最低4枚**（アイキャッチ1＋本文3）。見出しセクション数に応じて5枚以上でもよい

## 手順

### 1. 記事テキストの受け取り
- ユーザーから記事テキスト（タイトル・本文・タグ・カテゴリ・メタディスクリプション）を受け取る
- 記事のテーマ・キーワードを把握する
- **日付の検証**：検索・プロンプト実行の取り違えで**1年前の記事**が混ざることがある。本文・出典の掲載日が「今回のニュース」として妥当か確認する（`article_creation_guidelines.md` セクション1「日付・時系列の検証」、`.cursor/rules/article-creation.mdc` の「日付・時系列の検証（Logic）」参照）。

### 2. AI画像生成（実写で足りない枚・セクション用）
- 公式・一次画像で**各セクションに即した素材**が揃ったら、AI生成は**不足分のみ**。揃わない場合は `generate_image` または **Nano Banana 系・`GenerateImage`** で、**セクションごとにプロンプトを分け**、img_002以降は**その見出しの固有名詞・状況**をプロンプトに含める（汎用抽象画にしない）。
- **最低4枚**（多い場合は5〜8枚）を満たす（並行実行可）
- 生成した画像を `/Users/rinmon/Library/CloudStorage/SynologyDrive-work/0000/BLOG/drafts/` にコピー
  - ファイル名形式: `img_001_xxxx.png`, `img_002_xxxx.png`, ...
  - ソートされた順で最初のファイルがアイキャッチになるため、`img_001` を最も重要な画像にする

### 3. 記事ファイルの作成
- `drafts/` フォルダに `.md` ファイルとして記事を保存
  - 1行目: タイトル
  - 本文中に画像を `article_creation_guidelines.md` で指定された共通専用フォーマット **`[IMAGE_BLOCK]`** 形式で挿入する（アイキャッチも含めること）。
  - **【重要】アイキャッチは可能な限り Official・一次情報の写真を入れる。**本文2枚目以降も各セクションに即した画像とし、**汎用OGP・アイコン・バナーで枚数を埋めない。**（プレス・公式Xの `pbs.twimg.com` 等を優先し、`出典名`・`出典URL`・`キャプション`を正しく埋める）。
  - セクションに合う実写が無い場合は **Nano Banana／GenerateImage 等でその節向けに生成**し、IMAGE_BLOCK の `キャプション:` または `説明:` に「※画像は生成AIにより作成されたイメージです」と必ず付記し、`取得区分: 生成AIイメージ`。URLには `/Users/rinmon/.../BLOG/drafts/img_00X_xxx.png` のようなローカルパスを指定する。
  - 末尾に `**タグ**`, `**カテゴリ**`, `**メタディスクリプション**` セクション

### 4. 重複チェック（必須）

投稿する前に、**ローカル（他の .md）と WordPress（同一タイトル）** の二重を避ける。

- **今回のドラフト1本だけを検査**（推奨）: `python3 check_article_duplicates.py --file <作成したドラフト.md> [--scheduled]`
  - `--scheduled` を付けると、予約投稿（future）同士の同一タイトル重複も続けて表示する（`list_duplicate_scheduled.py` 相当）。
- **貯まったドラフト全体を棚卸し**: `python3 check_article_duplicates.py --scan-drafts`
- **Logic（エージェント側）**: 類似テーマ・類似タイトルの既存記事がないか、`processed/` やサイト内既報を軽く意識してからタイトルを確定する。

重複が解消できたことを確認してから次へ進む。

### 5. 投稿スクリプト実行

重複チェック通過後、**`--draft` は付けない**（既定で投稿予約 `future`）。**障害がなければ**必ずここまで実行し、ドラフト作成だけで終了しない。

```bash
cd /Users/rinmon/Library/CloudStorage/SynologyDrive-work/0000/BLOG && python3 api_poster.py --file /絶対パス/作成したドラフト.md
```

`--site chotto` 等でサイト別 `drafts/` が使われる場合、`--file` に **絶対パス**を渡すと確実。

### 6. 結果確認
- 投稿成功時: **タイトル全文**・Post ID・スケジュール日時（予約の場合）を確認しユーザーに報告する。**タイトルを省略してはならない**（スクリプト末尾の `📌 投稿サマリー` ブロックに同一のタイトルが出る）。
- 投稿失敗時: エラーメッセージを確認して対処
